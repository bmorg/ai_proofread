# CLAUDE.md — ai_proofread

Guidance for Claude Code (and contributors) working in this extension. It covers what's specific and non-obvious; the conventions this extension follows are listed at the end.

## What this is

A TYPO3 backend extension that runs **AI proofreading of page content**. An editor (or a bulk job) sends a page's text to an LLM and gets back a structured report of issues. It is **advisory only — it never blocks saving or publishing.**

- **Composer:** `bmorg/ai_proofread` · **extension key:** `ai_proofread` · **namespace:** `Bmorg\AiProofread\`
- **Targets:** TYPO3 **v11.5 / v12.4 / v13.4**, PHP **8.2+**. **v11 is strictly required** — it's the one place cross-version glue costs us (legacy module registration; see below). PHP 8.2 is the common floor across v11–v13.
- Publicly published, so favor no-code admin configuration and a clean public extension surface.

## Status (read before assuming things work)

This is an **early alpha**, in production use on **TYPO3 v11** — the only version it's actually been run on. **v12/v13 are code-targeted but not yet exercised on a live instance**: the dev loop here is the test harness (unit + functional on a real TYPO3 **v11**, see *Verify changes* below) plus `php -l` — nothing runs against v12/13. The things most likely to need attention on v12/v13:

- Backend **module registration** is split: `Configuration/Backend/Modules.php` (v12/13) and legacy `addModule` in `ext_tables.php` (v11). **Rendering is two-layer:** inner views via `StandaloneView::setTemplatePathAndFilename()` (absolute `EXT:` path — version-neutral, fine everywhere), and the module **chrome** via the version-guarded `ReviewModuleController::moduleResponse()` — v11/12 use `setContent()`/`renderContent()`, v13 (where those were removed) uses `assignMultiple(['content'=>…])` + `renderResponse('ModuleBody')`. The v13 path resolves `Resources/Private/Templates/ModuleBody.html` through the module route's `packageName` option (the backend-view convention is plain `Resources/Private/Templates/`, **not** a `Backend/` subdir — verified against v13 core source, but **not** yet on a live v13; this template resolution is the single most important thing to confirm there).
- **Run / mark / report / apply all live in the KI-Lektorat module** (no modal). Run/mark are tokenized GET links on the docheader (`?id=<page>&action=run|mark`); the review-and-fix queue's apply/dismiss/manual/reopen are tokenized GET actions too (`action=apply|dismiss|manual|reopen&reportUid=&finding=<index>`). **The whole flow works server-rendered (PRG); the *only* JS is one optional inline `<script type="module">` in `Review/Current.html`** that turns those actions into in-place updates (fetch with `X-Requested-With` → JSON), degrading to full-page reload when absent — see *Review-and-fix queue* below.
- **Entry point** is a "KI-Lektorat" docheader button injected into the **Page** module (`web_layout`) by `EventListener\AddProofreadButton`. Two mechanisms, version-guarded so they never double-add: **v11 uses the legacy `getButtonsHook`** (`SC_OPTIONS['Backend\Template\Components\ButtonBar']['getButtonsHook']`, registered in `ext_localconf.php`, method `getButtons`); **v12/13 use the PSR-14 `ModifyButtonBarEvent`** (`__invoke`, tagged in `Services.yaml`). v11 was originally event-only and the button never showed — that's why both exist now. The button's icon is **state-aware** (`decorationFor()` — a dedicated robot SVG variant + tooltip per page state, precedence queue-beats-review: pending/running → `aiproofread-robot-clock`, error → `aiproofread-robot-warning`, else Geprüft → `aiproofread-robot-check`, Report erstellt → `aiproofread-robot-info`, Ungeprüft → plain `aiproofread-robot`; registered in `Configuration/Icons.php`. Core icon overlays were tried first and rejected: the badge sat over the figure, got clipped by core's `.icon { overflow: hidden }` when nudged, and read poorly at docheader size). Because the v11 hook instantiates the class bare, it stays **constructor-less** — the repositories are resolved lazily in `buildButton()` (makeInstance ConnectionPool + `new`), like its UriBuilder/IconFactory use.

Scheduler/CLI commands: `bin/typo3 aiproofread:process-queue` (drains the module's run queue — register as a recurring Scheduler task) and `bin/typo3 aiproofread:subtree <rootPageUid>` (bulk). **The subtree bulk command is entirely untested** — no automated coverage, and unlike the queue path it hasn't been exercised in production — and has a known flaw: a page whose run throws aborts the whole command (no per-page catch — the remaining pages are skipped). Fix direction: per-page catch-report-continue, like `ProcessQueueCommand` does per job.

## Architecture

### LLM backend

**Scope is deliberately narrow: one backend — OpenRouter, via the OpenAI-compatible wire format.** (The native Anthropic `ClaudeClient` and a multi-backend selection mechanism existed earlier and were removed — we only use Claude *through OpenRouter*, and there were no resources to test anything else.)

`Service\Llm\LlmClientInterface` has `complete(...): LlmResult` + `completeBatch(requests, concurrency): list<LlmResult|LlmException>` (concurrent batch — see below). Two implementations:
- **`OpenAiCompatibleClient`** — the real client. OpenAI `/chat/completions`; with OpenRouter it also covers any OpenAI-compatible endpoint, but OpenRouter is what's configured/tested. Config splits in two: `baseUrl` (default `https://openrouter.ai/api/v1`) and `apiKey` (`sk-or-…`) are ext config (the **`provider`** tab); the model-shaping params — `model` (e.g. `anthropic/claude-opus-4.8`), `structuredOutput` (`json_schema`/`json_object`/`prompt`), `reasoning`, `maxTokens`, `pinProvider` — come from the **active model preset** (see *Model presets* below), read through the same `config()`/`ExtensionSettings` accessor. Auth is a fixed `Authorization: Bearer <key>` (the configurable auth-header/scheme/extra-headers knobs were dropped — speculative for OpenRouter-only). Extends `AbstractHttpLlmClient`, which owns transport, the **concurrent Guzzle pool**, single-call `complete()` (= a batch of 1), and error→`LlmException` mapping; the client just implements `buildCall()`/`parseResponse()`/`requestTimeout()`.
- **`MockClient`** — deterministic fake report, no API/key/cost; implements the interface directly.

`LlmClientFactory::create()` returns the mock when the **`useMock`** setting is on, else the OpenAI-compatible client; bound to the interface via `factory:` in `Services.yaml` (the binding keeps `class:` to satisfy autoconfigure). Constructors stay side-effect-free; validate config in `buildCall()`. On failure every client throws `LlmException` carrying the attempted request + raw response, so failures are auditable.

**Prompt design (`ProofreadingService::buildSystemPrompt`):** one prompt, model-agnostic (not model-specific — that rots). It combines an explicit error-type checklist (recall) with strict routing (precision): clear orthographic/grammar/punctuation errors **always** go in `findings`; only interpretive/stylistic doubt goes to `other`; `other` is explicitly liberal (style, content, currency notes). Plus "don't change meaning", "correct idiom isn't an error", and gender-inclusive language restraint (don't impose gendered forms; flag inconsistency in `pageFindings`). **Keep the default general** — resist baking in narrow, single-FP-derived clauses. That is exactly what `extraPromptInstructions` is for.

**Site context** is configurable so nothing site-specific is baked into the default prompt (the extension is publishable): the **`siteDescription`** setting fills the opening line (e.g. *"…für die Inhalte folgender Website: eine Fanseite für Klemmbaustein-Sets."*); empty → a fully generic *"…für Website-Inhalte."* with no domain reference.

**Site-specific rules** live in the `extraPromptInstructions` setting, **appended** to the prompt so they can't break the schema/category contract. This is the home for narrow per-site exceptions. Ready-to-paste examples: *headings/definitions may be sentence fragments* — `Sätze, die eine Überschrift beantworten oder einen Begriff definieren, dürfen Satzfragmente sein.`; *stylistic restraint* — `Sei bei Stilfragen besonders zurückhaltend.` Watch the first per-element runs for fragment false positives (an element is often a heading + short answer seen in isolation); if they appear, the fragment rule above is the fix. The full model/prompt evaluation — capability tiers, failure modes, prompt iterations, the scope finding, the test harness/outputs — lives in `model-experiments/README.md` (dev-only; not shipped).

**Quality / robustness (learned from testing):**
- **Reasoning** is the biggest quality lever — `reasoning` adds `{"reasoning":{"enabled":true}}` (OpenRouter); with it + the three-bucket prompt, Opus 4.8 matches/exceeds claude.ai web output. Needs a reasoning-class model; flash/mini models under-perform and some open models hallucinate suggestions.
- **Truncation detection is mandatory.** Reasoning tokens count against `max_tokens`; a verbose reasoner can exhaust the budget and — because strict structured output **force-closes the JSON** — emit a *parseable but near-empty* answer. So the client checks the **finish reason** (`finish_reason: length` / `native_finish_reason: max_tokens`) and throws `LlmException` — never trust "the JSON parsed". Without this, reasoning silently produces empty reports (observed: Sonnet 4.6 even at 30k).
- **OpenRouter can route to a provider that ignores `response_format`.** Then you get the JSON wrapped in ```json fences, or even *invalid* JSON (unescaped `"` in `quote`/`suggestion`, since no enforcing decoder escaped it) — observed with `"provider": "Google"` (Vertex) serving Claude. Two defences: (1) the **`pinProvider`** preset key → OpenRouter `provider: {order: [<slug>], allow_fallbacks: false}` pins the request to one provider that enforces valid JSON (e.g. `anthropic`); empty → free OpenRouter routing (no `provider` block). `allow_fallbacks` must be snake_case + a real bool (a camelCase key or `"false"` string is silently ignored → fallbacks stay on). (`pinProvider` replaced an earlier `providerOrder`+`allowFallbacks` pair — they were only ever set as pin-exactly-one-no-fallbacks or none, so they collapsed to one key.) Two pitfalls learned the hard way: don't use `require_parameters` (Claude providers don't advertise native `response_format` support, so it excludes them all → 404); and the **model slug must be one Anthropic actually serves** (`anthropic/claude-opus-4` is a *dead* slug — only a Google passthrough served it; with fallbacks off, pinning `anthropic` to it 404s. Use the current `anthropic/claude-opus-4.8`). (2) parsing is **tolerant** (`tryParseJson`: raw, else first `{…}` span) to strip fences/prose. A genuinely malformed reply still errors (no silent empty).
- **Straight `"` inside prose fields truncates them.** Under strict structured output the constrained decoder treats a literal `"` as the end of the JSON string — so when the model uses a straight quote as a *typographic* quotation mark in `explanation`/`observation`/`other` (e.g. `„Sie"`), the value is silently cut off there (valid JSON, short text; `finish_reason: stop`, so truncation detection won't catch it — it's not output truncation). Not a parse/merge/display bug; the text is genuinely lost in the model output. Mitigation is in the prompt: instruct German typographic quotes `„…“` and **never** a straight `"` in the free-text fields. (The `quote`/`suggestion` fields are unaffected — there the model escapes `\"` correctly.)
- **`max_tokens` is a preset key** (`maxTokens`, 0 = auto). Auto-default jumps to 32000 when reasoning is on.
- **Request timeout is configurable** (`requestTimeout`, 0 = auto). Auto is 180s, or 600s for the OpenAI client when reasoning is on (reasoning models are slow — minutes). Runs happen in the **`aiproofread:process-queue` CLI/Scheduler** context (not the web request — see "Async run queue"). A page's N+1 passes run **concurrently** (`maxConcurrency`), so per-page wall-clock is ~one slow call; but the command may process several pages in a row, so still give the PHP CLI a generous `max_execution_time`.

**Decisions not to re-litigate:**
- **One backend only (OpenRouter / OpenAI-compatible).** A native Anthropic client + a multi-backend factory (tagged-iterator selection by a `backend` setting) were built and then removed: we use Claude *via OpenRouter*, and there were no resources to test other providers. Don't re-add provider plumbing speculatively — the interface + `AbstractHttpLlmClient` are enough to grow back into if a second backend ever earns its keep.
- The bundled **`claude-api` skill is Anthropic-native (SDK/Messages API)** — do **not** use it for this extension's OpenRouter/OpenAI-compatible HTTP; verify the chat-completions schema against current OpenRouter/OpenAI docs instead.

### Model presets

The model and its coupled params are **not** ext-config knobs — they live in a curated enum **`Enum\ModelPreset`** (cases like `Opus48`, `Sonnet46`, `GLM52`; each `->settings()` returns the bundle: `model`, `reasoning`, `maxTokens`, `structuredOutput`, `pinProvider`). One exception: the **`Custom`** case pins nothing in the enum (`settings() => []`) — its five params are admin-entered and stored at runtime (below), so the picker isn't strictly curated-only. One preset is active **site-wide**, held in `sys_registry` via **`Service\ActivePreset`** (`current()`/`set()`; default `ModelPreset::default()` = Opus 4.8 with reasoning) — mutable at runtime, **no cache flush, no table, no TCA**. `ActivePreset` also owns the **custom values** (a second `sys_registry` key): `customSettings()`/`setCustom()` (both `normalize()` the 5 keys — whitelist `structuredOutput`, clamp `maxTokens ≥ 0`, cast `reasoning`; default to the default preset's settings when unset) and **`effectiveSettings()`** — the active preset's `settings()`, or the custom values when `Custom` is active.

The overlay is a single chokepoint: **`ExtensionSettings::get($key)`** consults **`ActivePreset::effectiveSettings()`** (so the `Custom` branch lives entirely in `ActivePreset` — `ExtensionSettings` and the client stay preset-agnostic), returns that value when it pins `$key`, else falls through to ext config. So every consumer (`OpenAiCompatibleClient`, `ProofreadingService`, the factory) reads through the same `config()` and is preset-aware with **no preset-specific code**. `useMock` is deliberately *not* a preset key (it flips the container-build factory binding → would need a cache flush).

Admins switch the active preset — and edit the custom config — in the module's **Einstellungen** view (admin-only). It's a radio list; each row shows the model slug + its params as chips (`summarizeSettings()` → `{model, chips[]}`), with the active row highlighted. The **`Custom` row carries the five input fields inline**, revealed only when its radio is the current selection via a CSS **`:has()`** rule (no JS — reacts to the *unsaved* choice; `@supports not selector(:has(*))` falls back to always-shown). Saving is a no-JS POST → PRG; the custom fields are persisted on **every** submit (so edits survive switching presets and apply the moment `Custom` is picked), and a **`FlashMessage`** confirms the save across the redirect. **Adding a *shipped* model = adding an enum case** (the `Custom` case is the admin-defined escape hatch).

**Why / decisions not to re-litigate:** swapping models meant changing several coupled keys at once and remembering how to revert — a named bundle fixes both. The roster is **curated** plus a single **`Custom`** admin-defined slot (no arbitrary user-named presets), **global** (one active for the whole install — an admin-once setting, *not* per-editor or per-run), and set in a **settings view, not the per-page docheader** (a docheader dropdown was built first and rejected: it put a rare admin action into every editor's per-page flow). Method is `settings()` (not `overrides()`): after the keys left ext config, a preset is the *source of truth* for them, not an override on top.

### Scope: split per request (N+1), merged into one page report

A check is **page-scoped in result but split in execution** (`ProofreadingService::proofreadPage`). Two scopes, because they want opposite things:

- **`findings` (localized) — one focused LLM pass per content element.** Whole-page input *dilutes attention*: a capable model glides past a subtle local error it catches when focused on the single element (proven empirically — a stray subject–predicate comma missed whole-page, caught on the lone paragraph; *same prompt*, only scope changed. See `model-experiments/README.md` → "input scope"). Prompt wording does **not** fix this; scope does.
- **`pageFindings` + `other` (page-wide) — one whole-page pass** (`extractPageText()`), which preserves the cross-element context those buckets need (consistency, currency). Its `findings` are discarded — the per-element passes own those.

So a run is **N+1 LLM requests** (N elements + 1 whole-page), merged into one three-bucket report. Same prompt/schema for every pass; the caller keeps the relevant buckets (element passes → `findings`; whole-page pass → `pageFindings`/`other`, its findings discarded). The passes are independent and I/O-bound, so `proofreadPage` fires them **concurrently** via `LlmClientInterface::completeBatch`, bounded by the `maxConcurrency` setting (default 4 — a provider rate-limit guard) — wall-clock is ~one slow call, not the sum. Each call is logged individually; one failed pass fails the whole run. Cost/latency is still why a run is **queued, not inline** (see below). Each per-element pass's findings are tagged with `elementUid`/`elementLabel` for the review-and-fix queue (see below), but the report stays **rendered flat** — grouped by category, no per-element sections.

The three buckets (`buildReport`, rendered in `Review/Current.html`): `findings` (localized, `quote→suggestion`), `pageFindings` (page-wide structured), `other` (free-text catch-all). Reaching web-chat quality needed all three plus reasoning — see below.

### Async run queue (Scheduler)

Because a run is N+1 reasoning requests (concurrent, but still minutes of wall-clock), the module **never proofreads inline**. "Report erstellen" enqueues a job in `tx_aiproofread_queue` (`QueueRepository::enqueue`, deduped per page/language); the **`aiproofread:process-queue`** command (register as a recurring **Scheduler** task) drains it via `proofreadPage`. Job lifecycle: pending → running → (success: row deleted, the review snapshot is the result) / (failure: kept as `error` + message). The module shows **"Report wird erstellt …"** while pending/running and surfaces a failed job; `findForPage` drives that. **Ops requirement:** without the Scheduler task running, clicks queue but nothing processes.

**Stale-job reclaim uses a heartbeat, not `started_at`.** A dead worker (OOM/timeout/deploy) would leave a row stuck `running` forever; `reclaimStaleRunning` (start of each drain) flips rows stale beyond `STALE_RUNNING_SECONDS` (30 min) to `error` so the page un-wedges. Staleness is measured on the **heartbeat** (`tstamp`): the worker calls `QueueRepository::heartbeat($uid)` as each of the N+1 passes settles (via the `?callable $onProgress` threaded `ProcessQueueCommand` → `proofreadPage` → `LlmClientInterface::completeBatch`, fired from the Guzzle pool's per-request `fulfilled`/`rejected`). So a legitimately long run (many elements × slow reasoning calls, which can exceed 30 min wall-clock) keeps refreshing and is **not** reclaimed while alive — only a worker that stops settling passes goes stale. The gap between heartbeats is bounded by the longest single pass (`requestTimeout`, ≤600s), well under the threshold. `heartbeat()` is guarded on `status='running'` so it can't resurrect a finished/reclaimed job.

### Report history + status model

**Reports accumulate.** Each run is an **immutable row** in `tx_aiproofread_report` (`ReportRepository`): page, language, user, model, the merged report JSON, the `categories` enabled for that run (so the Verlauf can tell "checked, found 0" from "category disabled"), cost, duration, and a `content_hash` (metadata for the bulk command's skip-unchanged check only — *not* for status). The newest run for a page is its "Aktueller Report"; the full list is the "Report-Verlauf" (which decodes `report_json` for the per-category finding breakdown). No cross-run finding reconciliation — each run is just a frozen snapshot.

**Sign-off is page-level.** `tx_aiproofread_review` is now just per-page sign-off state (`proofed_at`/`proofed_by`); marking "geprüft" records *when*, nothing else.

**Status is derived from timestamps — no content hash, no "veraltet"** (`ReviewRepository::deriveStatus`, `Enum\ReviewStatus`):
- no run → **Ungeprüft**; signed off at/after the latest run → **Geprüft**; otherwise (a run newer than the sign-off, or never signed off) → **Report erstellt**.
- Rationale: the normal flow is *read report → edit page → mark geprüft*, so the content is **always** different afterward — tracking "changed since" would just nag. Re-running creates a newer run → the page naturally flips back to *Report erstellt*. (The hash + a "veraltet" state were removed; the hash lives on for the bulk skip only.)

### Review-and-fix queue (apply / dismiss / deep-link)

Localized `findings` are no longer read-only — *Aktueller Report* is a worklist the editor acts on (the "make-or-break" feature: a read-only report no one acts on is worthless). This **reversed three earlier "deliberately not built" decisions** (per-element attribution, content write-back, and backend JS); the reversals are scoped tightly.

- **Per-element attribution (findings only).** `ProofreadingService::proofreadPage` now tags each `findings` item from a per-element pass with `elementUid` + `elementLabel` (the whole-page pass's `pageFindings`/`other` stay un-attributed — they describe cross-element patterns with no single anchor). The field the quote came from is **not** stored (the model sees concatenated text); the applier searches `header`/`subheader`/`bodytext`. `bodytext` qualifies only when the element's CType configures it as RTE (`enableRichtext` in TCA): plain-text bodytext (core `table`, `bullets`) would be corrupted by the DOM parse-and-reserialize, so it reports `UNSUPPORTED` → deep-link only (the guard runs at apply/locate time, so it also protects findings from old reports). The raw-markup `html` CType is excluded from extraction entirely (`ContentExtractor::EXCLUDED_CTYPES`) — it never produces findings. Old reports lack `elementUid` → their findings just show no apply/deep-link (graceful).
- **Apply = `Service\SuggestionApplier`.** Re-locates the finding's `quote` in the *current* stored field (the report is a snapshot, so content may have changed — a miss is expected, surfaced as a flash + deep-link, never a wrong edit). `header`/`subheader`: exact, unique substring. `bodytext`: parse the RTE HTML (`DOMDocument`), match against a **single text node**'s decoded text (apples-to-apples with what the model was fed); a quote that spans markup (`<strong>…`) or isn't unique is reported (`SPANS_MARKUP`/`AMBIGUOUS`) and left to the deep-link, never spliced. The write goes through **DataHandler run as the BE user** (the extension's *only* content mutation — every other write is direct ConnectionPool) so tt_content edit perms, RTE transformation and sys_history/undo all apply. Typed result: `applied`/`notFound`/`ambiguous`/`spansMarkup`/`unsupported`/`noPermission`/`error`. `SuggestionApplier::locate()` is also the render-time pre-check that decides whether *Übernehmen* shows + supplies the context snippet.
- **Per-finding state = `tx_aiproofread_finding_state` + `Service\FindingStateRepository`.** Reports are immutable, so the per-finding decision lives here, keyed by `(report_uid, finding_index)` — the index into that run's stored (already category-sorted) `findings` array. Absence = open; `accepted` = written back via apply; `manual` = the editor fixed it themselves (no write-back — the honest resolution for findings that aren't auto-applyable, e.g. spans-markup/ambiguous, or that they preferred to edit by hand); `dismissed` = rejected. `manual` and `dismissed` are reopenable; `accepted` is not (undo an applied change via TYPO3 record history). Progress ("x von y erledigt") counts all decided indices that still exist in the run.
- **Controller**: `ReviewModuleController` adds tokenized PRG actions `apply`/`dismiss`/`manual`/`reopen` (`handleFindingAction`; reopen refuses `accepted` server-side — the CSS-hidden link is not the only guard); with `X-Requested-With` they return JSON (new state + progress) for the inline JS, else flash + redirect. **Permissions are two-tier:** viewing needs page-show; every mutating action — run, mark, and all finding decisions — additionally requires content-edit on the page (`$canEdit` via `doesUserHaveAccess(…, Permission::CONTENT_EDIT)`); for read-only users the run/mark buttons and per-finding action links simply aren't rendered (empty URLs → hidden by the template), and a hand-crafted action URL gets a clean German refusal. `buildReportSections()` enriches each finding with `index`, `state`, `editUrl` (`record_edit` deep-link with `returnUrl`), `applyUrl`/`dismissUrl`/`manualUrl`/`reopenUrl`, `applicable`, and the context snippet.
- **UI** (`Review/Current.html`): each finding card shows the element label, the existing quote→suggestion diff, the context line, explanation, and the actions. State drives visibility via a `state-<status>` class on the card (server-rendered initial state *and* the JS-updated state share one set of CSS rules). Page-wide/free-text hints sit under a separate **"Erfordert Ihre Einschätzung"** advisory block (no apply — inherently un-anchorable). The optional inline module script swaps the state class + progress in place; no RequireJS, no build, **may be CSP-blocked on v12/13 → falls back to the working PRG path** (add to the v12/13 verification list).
- **Known edge — unknown categories.** In the non-strict output modes (`json_object`/`prompt`, or a provider ignoring `response_format`) the model can emit a `findings` category outside `Enum\Category`. Such findings are **stored** (`sortByCategory` keeps them, sorted last — honest snapshot), **not rendered** (`buildReportSections()` has no section for them), but **counted** by `findingProgress()` — so "alle erledigt" is then unreachable. Accepted; if it ever bites, add a trailing "Sonstige" fallback section rather than filtering the total. (Impossible under `json_schema`: the enum is enforced.)

### Removed / not built (deliberately)

- **Per-element attribution beyond `findings`.** `pageFindings`/`other` are still un-attributed (no single anchor), and the report has no per-element *sections* — findings are tagged with `elementUid` but still rendered grouped by category, not by element.
- **Reverting an applied suggestion in-module.** Dismiss is reopenable; an *applied* change is undone via TYPO3's normal record history (DataHandler writes `sys_history`), not a module action.
- **Multi-language support.** The *write* side threads `language_uid` through (`ProofreadingService`, the queue, the report/queue tables keep the column), but the **read** side is deliberately L0-only for 0.1: `ReportRepository::find*ByPage()`, `ReviewRepository`, and the controller's `enqueue(..., 0, ...)` ignore language, so the module surfaces only the default language. The `subtree` command's `--language` option was **removed** (0.1) because it wrote rows the read paths couldn't correctly surface. The columns stay so a future language-aware read pass doesn't need a migration.
- **Workspace support.** Out of scope, like multi-language. All reads (extraction, the applier's `locate()`, the module views) query **live records only** — an editor working in a workspace gets reports about the live page, and an apply goes through DataHandler in that user's workspace context (creating a workspace version of an element whose *live* content was located). There is no overlay handling and no guard; documented in the README as a limitation.

### Possible future improvements (discussed, not built)

- **Apply suggestions** — *done* (see *Review-and-fix queue*). Remaining edges: bodytext quotes that span markup or repeat aren't auto-applyable (deep-link only); a quote with whitespace drift from the stored field won't match (deep-link only).
- **Co-locate the report with the edit form** (findings panel inside the content element's FormEngine form). The deep-link is the lightweight stand-in for now.
- **Richer report in the module**: per-finding "copy" buttons and category filtering. *Done already:* char-level red/green diff (`ReviewModuleController::diffHighlight`), whitespace glyphs, and now per-finding apply/dismiss + element deep-links.
- **Inline accept/reject in the RTE** (CKEditor plugin, Grammarly-style) — best UX, most work, crowded lane (WProofreader); a poor fit for this batch/queued model.
- **Attributing `pageFindings` to elements** by feeding the whole-page pass element-labelled structured input, so page-wide hints could carry soft deep-links.
- **Report-Verlauf cap/pagination.** `reportsResponse()` loads and JSON-decodes **every** run of the page on each view (for the per-category counts) — reports are immutable and accumulate, so this view gets slower forever. Cap it (e.g. newest 50 + "alle anzeigen") or paginate; the decode-per-row is the cost, not the SELECT.
- **Memoize the per-finding locate() pre-check.** `buildReportSections()` calls `SuggestionApplier::locate()` for every still-open finding on each *Aktueller Report* render — one `tt_content` SELECT plus a full DOMDocument parse per finding, even when many findings share one element. Fine for the handful a page typically has; a per-element memo (load + parse each element once per render) makes finding-heavy pages cheap too.
- **Audit-log pruning.** `tx_aiproofread_log` grows without bound — every run stores the full page text N+1 times in `request_json` (plus the full responses), so on an active install this becomes the largest table in the DB. Planned: a `aiproofread:prune-log --older-than=<days>` command (Scheduler-schedulable, like the other two); possibly a lighter variant that only strips the bulky `request_json`/`response_json` from old rows and keeps the metadata (user/model/tokens/cost) for accounting.

### Categories

`Enum\Category` — fixed display order (`sort()`): **Rechtschreibung → Zeichensetzung → Grammatik → Gendern → Stil**. Drives report section order, the Report-Verlauf columns (`abbreviation()`: Rechtsch./Zeich./Gramm. — **gender-inclusive language *and* style are excluded** there: by prompt design neither is a localized finding (gender-inclusive language → pageFinding, style → the free-text `other` bucket), so their columns would always be 0 (style is already reflected in the Sonstiges count); a run that had a category disabled shows "–" for it, vs. "0" = checked/none), and the structured-output schema enum. Gendern and Stil are config-toggleable (both on by default; Gendern policy/form configurable). The Aktueller Report view shows whatever is in the stored report (all categories), regardless of current config.

### Audit log

Every LLM call (success or failure) is logged to `tx_aiproofread_log`: full request/response JSON, initiating BE user, model, **provider** (the actual OpenRouter provider that served it, from the response — useful for debugging routing), tokens, cost, **call duration (`duration_ms`)**, success/error, plus the **run (`report_uid`)** and **content element (`element_uid`/`element_label`)** it belongs to. Surfaced in the **API-Verlauf** view (list + detail). The queue runs a page as **N+1 concurrent calls**, so one "Report erstellen" produces N+1 log rows (one per element + one whole-page "Gesamte Seite"), each with its own duration — all attributed to the requesting BE user (the queue job carries `be_user`). Note: logs are *buffered* during the run and written only after the report row exists (so `report_uid` can be set); a failed run logs its calls with `report_uid = 0`. Cost is **USD** (`cost_usd`), taken from OpenRouter's reported `usage.cost` (the client sets `usage.include=true` to get it); 0 if a backend omits it.

### Backend modules

**One** module under Web — **KI-Lektorat** (`ReviewModuleController`), with a docheader **view-switcher dropdown** (`MenuRegistry`, `?view=…`) of five views (the last three admin-only):
- *Aktueller Report* (`current`): status + docheader actions (Report erstellen / Neuen Report erstellen, Als geprüft markieren, Seiteninhalt bearbeiten → `web_layout`) + the latest run's report as a **review-and-fix queue** (per-finding apply/dismiss/deep-link + progress — see *Review-and-fix queue*) in a `<details>` (**open** unless geprüft). `?reportUid=` opens a specific historical run (always expanded) — **honoured only if that run belongs to the current page** (else it falls back to the page's latest run), so a `?id=<ownPage>&reportUid=<otherPage's run>` can't disclose another page's report.
- *Report-Verlauf* (`reports`): the page's report runs (date · user · model · counts · cost); each row opens that run in *Aktueller Report*.
- *Statistik* (`stats`, **admin-only**): install-wide success metrics via `Service\StatisticsService` — coverage (content pages with a report / geprüft, against all live default-language `doktype=1` pages; hidden included) and finding outcomes. **"Behoben" = cumulative `accepted`+`manual` decisions across ALL runs** (decisions persist per run, so fixes stay credited after the errors vanish from newer reports; fixes made outside the review-and-fix queue leave no row and are *not* counted — accepted trade-off), "Verworfen" = `dismissed`, "Offen" = undecided findings of each live content page's latest run, by category. Ignores the page context like Einstellungen. Perf: decodes one `report_json` per checked page per render — fine admin-only; denormalize a count column if it ever hurts.
- *API-Verlauf* (`history`, **admin-only**): the per-call audit log (technical; not for editors).
- *Einstellungen* (`settings`, **admin-only**): the site-wide active model preset **+ the inline custom-model config** — radio list with per-preset chips and the active row highlighted, saved by a no-JS POST → PRG with a `FlashMessage` confirmation (see *Model presets*).
- **Default landing depends on status:** no explicit `?view=` → geprüft pages open on *Report-Verlauf*, others on *Aktueller Report*. Every dropdown item carries an explicit `view=` so a manual pick overrides that default. Run/mark/preset-save are tokenized actions that **redirect (PRG)** to the clean URL. A non-admin requesting an admin-only view (stale/hand-crafted URL) falls back to *Aktueller Report*.

Registered in `Configuration/Backend/Modules.php` (v12/13) and `ext_tables.php` (v11 fallback). All in-module links are built with `UriBuilder` (tokenized) — a bare `?logUid=…` href would drop the module token *and* the view.

## Conventions specific here

- Config lives in `ext_conf_template.txt`, flat camelCase keys grouped by `# cat=` (`backend`, `provider`, `categories`) — but the **model-shaping params live in `Enum\ModelPreset`, not here** (see *Model presets*); ext config keeps `baseUrl`/`apiKey`, timeouts/concurrency, `useMock`, and the prompt/category settings. Reads go through the injectable **`ExtensionSettings::get()`** service, which overlays the active preset then, on a missing key, returns its `DEFAULTS` entry (for keys whose template default is **not** the type's zero value — currently `baseUrl`, `genderInclusiveStyle`, and the on-by-default categories `enableStyle`/`enableGenderInclusiveLanguage` — mirroring `ext_conf_template.txt`; a type-zero default needs no entry, the inline `?? 0`/`?? ''` reproduces it) or null so callers default; each consumer keeps a thin `config()` that delegates to it. (TYPO3 does not apply `ext_conf_template.txt` defaults at read time — only on install-sync/save — hence the runtime mirror.) The `Service\PageTreeService` (QueryBuilder BFS, permission-filtered) resolves subtree page UIDs for the bulk `aiproofread:subtree` command (its only consumer) — it replaced the `@internal` `PageTreeView`.
- **Toggling `useMock` requires a TYPO3 cache flush** — the factory's interface binding is resolved at container-build time (true of any Services.yaml-driven behavior). The active **model preset does *not*** — it's read at runtime from `sys_registry`. **Changing `Configuration/Icons.php` needs a system-cache flush too** (the icon registry is cached in the `assets` cache, system group; without the flush new identifiers render as icon-default-not-found).
- **`Permission` is `Type\Bitmask\Permission`**, not the removed `Type\Bit\Permission` (gone in v12) — the portable path across v11.5/v12/v13.
- **`FlashMessage` severity is version-split** (v11 `int` constant `AbstractMessage::OK` vs v13 `ContextualFeedbackSeverity` enum — each removed on the opposite version, so neither is namable on both). The Einstellungen save-confirmation sidesteps it by **omitting severity** and using the named arg: `new FlashMessage('…', '', storeInSession: true)` — defaults to OK, version-neutral. Enqueued on the default queue (`FlashMessageService::getMessageQueueByIdentifier()`); `storeInSession` carries it across the PRG redirect, where the module chrome renders it — automatically on the v11/12 `renderContent()` path, and explicitly on v13 via `<f:flashMessages queueIdentifier="core.template.flashMessages" />` in `ModuleBody.html` (the custom v13 wrapper renders no flash queue otherwise). **Still unverified on a live v12/13 — see test checklist.**
- Tables: `tx_aiproofread_report` (report history), `tx_aiproofread_review` (per-page sign-off), `tx_aiproofread_log` (per-call audit), `tx_aiproofread_queue` (run queue), `tx_aiproofread_finding_state` (per-finding decision — accepted/manual/dismissed — for the review-and-fix queue, keyed by `report_uid`+`finding_index`). No TCA (accessed via QueryBuilder only). The active preset is a single value in core `sys_registry` (no own table).

## General conventions

This is a TYPO3 CMS extension. Frontend/UI language is **German**; code, comments and identifiers are **English**.

- **PHP 8.2+**, `declare(strict_types=1)` in every file.
- **Constructor injection** for dependencies — no `GeneralUtility::makeInstance()` for injectable services, no `inject*` methods. Register listeners/tagged services in `Configuration/Services.yaml`. (Documented exception: `AddProofreadButton` stays constructor-less because the v11 hook instantiates it bare — see *Status*.)
- Prefer **QueryBuilder** over raw SQL; use **`EXT:`** path syntax for resources; do not use `TYPO3\CMS\Extbase\Domain\Model\FrontendUser` (removed in v12).
- Enums (PHP 8.1+) are used for fixed vocabularies (`Enum\Category`, `Enum\ReviewStatus`, `Enum\ModelPreset`).
- **Server-rendered first; JS is optional enhancement only.** The UI is server-rendered Fluid + docheader links, and every flow (including apply/dismiss) works with no JS via tokenized PRG actions. The single exception is one inline `<script type="module">` in `Review/Current.html` that makes the review-and-fix queue update in place; it degrades to full-page reload (and may be CSP-blocked on v12/13). **No RequireJS, no build step, no separate JS asset** — keep it that way (the earlier context-menu/modal JS + the v11 RequireJS cache-busting headache are why).
- **Verify changes with the test harness** (plus `php -l` for quick syntax checks):
  - `composer install` once, then `composer test:unit` (milliseconds; extraction, applier matching truth table incl. the non-RTE corruption guard, prompt building, OpenRouter request/response handling, the docheader-icon state truth table) and `composer test:functional` (real TYPO3 booted into `.Build/` on **sqlite** — no DB server needed; real TCA/DataHandler: apply write path, sys_history, permission refusal, report page-ownership/disclosure regression, queue lifecycle/race guarantees, the Statistik coverage/outcome math). `composer test` runs both.
  - The dev install pins **TYPO3 v11** (production parity) via `typo3/testing-framework ^6`. A v12/v13 run (`composer update --with typo3/cms-core:^12.4` / `^13.4` + testing-framework `^7`/`^8`) is **not yet a working one-liner**: TF 7/8 type their `FunctionalTestCase` properties (`protected array $coreExtensionsToLoad`), and the functional test classes here declare them untyped (TF6 style) — PHP fatals on redeclaring a typed property untyped, so those classes need version-appropriate property declarations first. The unit suite should run unchanged (PHPUnit-9-style `test*` methods). Untested either way — treat the first re-pin as its own small task, not a config flip. `composer.lock` and `.Build/` are gitignored.
  - **Parallels/fuse caveat (this dev box):** `.Build` must live on the VM's native filesystem — it is a symlink to `/home/vagrant/.build-cache/ai_proofread`. On the `prl_fsd` shared folder, `realpath()` inside the functional-test instance drops the `/projects` mount prefix and the instance's core symlinks break. If `.Build` is ever recreated as a plain directory, functional tests fail with `symlink(): No such file or directory` / "typo3/sysext/*/ does not exist" — restore the symlink. Because `.Build` is a symlink, the extension is linked into the test web dir with an **absolute** target by `Build/Scripts/ExtensionTestLink.php` (composer `post-autoload-dump`; cwd-independent, paths derived from composer config) — a relative link resolves against the link's real parent chain, points into the build cache, and sends the classic-mode classmap scan through core+vendor via a symlink cycle (20+ min boot, no error).
  - Testability seams are marked `@internal`: `SuggestionApplier::analyze()`, `ProofreadingService::buildSystemPrompt()`/`buildSchema()`, `OpenAiCompatibleClient::buildCall()`/`parseResponse()`, `AddProofreadButton::decorationFor()` are public **only** for unit tests. Final services are never mocked — tests build the real object graph on mocked core deps (`ConnectionPool`, `Registry`, `ExtensionConfiguration`).
  - Live-instance checks remain for what the harness can't reach: module chrome on v12/13, backend CSP vs. the inline script, real OpenRouter routing, CKEditor serialization.
